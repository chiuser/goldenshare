# AGENTS.md — frontend 前端治理规则

## 适用范围

本文件适用于 `frontend/` 目录及其所有子目录。

若未来更深层目录存在更具体的 `AGENTS.md`，应同时遵守；发生冲突时，以更靠近当前目录的规则为准。

---

## 当前阶段说明

当前前端处于“任务中心试点、门禁深化与第一轮规模化推广已完成”的治理收敛阶段。
当前默认不再把“继续开推广批次”当成默认动作，而是先按 `Phase 6` 收口总结与残留清单决定下一步。

已有基础：

- React 19 + TypeScript + Vite
- Mantine 作为主 UI 基座
- TanStack Query / Router
- 第一批共享组件标准件与领域组件最小版本
- 任务中心试点页链路
- HTML Showcase 视觉对照
- `typecheck + test + build + smoke/visual gate` 基础门禁
- `Frontend Quality Gate`

当前主要问题不是“没有前端”，而是：

1. 缺少完整的前端交付流程
2. 缺少统一的交互与视觉 token 规则
3. 已有共享组件与试点页成果还需要更稳的门禁保护
4. 缺少更多可执行的前端规则自动检查
5. `DataTable v1` 与 `TradeDateField v2` 这类试点基线仍需要在后续推广中守住边界

因此，当前阶段的默认目标是：

- 优先维护流程、规范、组件标准和测试门禁
- 保护试点页和共享标准件已经形成的稳定基线
- 深化 smoke / visual gate 与规则自动检查
- 在新页面和被修改页面中逐步收敛风格
- 不做一次性大改全部页面的 big bang 重写

## 当前硬约束

### 1. Mantine 仍是唯一主 UI 基座

- 不引入第二套主组件体系
- 不在新页面中混入另一套全局设计语言
- 若 Mantine 默认行为不满足需求，优先在主题层或共享组件层收敛

### 2. 新 UI 不再继续放大旧的紫色渐变 / 玻璃拟态风格

当前仓库里仍存在旧视觉遗留，例如：

- 紫色系品牌色
- 渐变背景
- `glass-card`
- 偏展示型阴影

这些遗留暂不要求一次性清空，但从本规则生效开始：

- 新页面默认收敛到“中性灰 + 深海军蓝 + 低阴影 + 明确边框”的方向
- 被大幅修改的旧页面，应顺手收敛到新标准
- 纯 bugfix 不要求强行重做整页视觉

### 3. 共享组件优先于页面内临时拼装

重复出现两次以上的模式，优先抽到共享层，而不是复制第三份。

典型模式包括：

- 页面头部
- Section 卡片
- 状态标签
- 空状态
- 日期选择
- 筛选栏
- 表格壳
- 抽屉详情
- 操作反馈

### 4. 语义状态与涨跌颜色必须分离

本平台默认遵循 A 股语义：

- 红涨绿跌

但系统反馈语义仍然独立维护：

- success / warning / error / info

禁止把“上涨”直接映射成 success，把“下跌”直接映射成 error。

### 5. 前端不为后端临时对象命名背锅

前端页面必须优先使用用户任务语言，不要把内部对象直接暴露为主阅读路径。

例如：

- 不要把 `execution`、`job_name` 或内部路由 key 直接当页面标题
- 不要把后端字段名直接作为表单 label
- 不要让用户理解页面必须依赖源码知识

任务中心当前强约束：

- 任务记录、任务详情、重试、停止统一消费 TaskRun API：`/api/v1/ops/task-runs*`
- 手动维护提交统一消费 `/api/v1/ops/manual-actions/{action_key}/task-runs`
- 不得恢复旧 `/api/v1/ops/executions*`、`steps/events/logs` 或 `sync_run_log` 作为页面事实源
- 不得把 `sync_daily / backfill_* / sync_history` 暴露为用户需要选择或理解的任务类型

### 6. 不要为前端任务顺手触碰后端重构主线

前端任务默认不跨到：

- `src/foundation/**`
- `src/ops/**`
- `src/operations/**`
- `src/platform/**`
- `src/app/**`

除非任务明确要求，或页面改动必须伴随后端契约变更。

### 7. 试点支持任务也必须守住单一主目标

以下能力虽然已经在 `Phase 4` 中落下，但在 `Phase 5/6` 继续演进时，仍然必须按“单一主目标 + 边界卡”推进：

- `DataTable v1`
- `TradeDateField v2`

不要因为它们属于基础设施支持任务，就顺手扩大到无关页面或无关能力。

## 页面与组件分层规则

### `src/app/**`

只放：

- Provider
- Router
- Shell
- Theme
- 全局应用装配

不要把页面业务逻辑继续堆进 `app`。

### `src/features/**`

放领域级状态、hook、上下文、流程逻辑。

### `src/shared/api/**`

只放请求客户端、错误、API 类型和通用 transport 逻辑。

不要在这里放页面拼装逻辑。

### `src/shared/ui/**`

放跨页面共享的 UI 组件与交互模式。

要求：

- 尽量纯展示
- props 语义明确
- 不绑定具体页面接口路径
- 能单独测试的优先补测试

### `src/pages/**`

页面负责：

- 页面级查询组合
- 页面状态流转
- 共享组件编排

页面不应继续无限变厚。

当出现以下信号时，优先拆分：

- 单文件明显超过 400-600 行
- 同一页同时承载筛选、表格、详情、表单、状态轮询
- 出现多个可复用区块

## 前端任务工作方式

## 技术栈流程门禁（新增）

基于当前技术栈（React 19 + TypeScript + Vite + TanStack Router/Query + Mantine），新增以下流程门禁：

1. 数据获取必须通过 `shared/api` + Query 层，不允许在页面组件内直接拼接后端返回字段形成业务结论。
2. 页面渲染必须消费“已映射 ViewModel”，禁止页面层临时做 DTO 二次拼装。
3. 不接受兼容方案和临时方案：若接口契约不合理，先提契约改造方案再开发，不允许“先糊一层页面兼容”。
4. 任何新增页面或中等以上改动，开发前必须有详细技术方案并完成评审，评审通过后再进入编码。
5. bug 修复必须先说明根因与合理性，不允许只做表层现象修补；若修复会扩散，应优先做结构化修复。

---

## 代码规模与组件化门禁（新增）

为避免前端代码堆砌，新增以下硬性要求：

1. 单文件超过 `400` 行时必须评估拆分；超过 `600` 行默认视为超长文件，必须拆分并在 PR 说明拆分方案。
2. 页面文件不直接承载复杂业务编排、表格列定义、表单规则、详情渲染全部逻辑；应拆到 `features/shared/ui`。
3. 对出现三次以上的重复 UI/交互模式，必须抽象沉淀为共享组件或共享 hook。
4. 拒绝“复制一份改一改”模式；必须给出复用点与抽象边界。

---

## 架构优化触发条件（新增）

满足以下任意条件时，必须提出架构优化建议，不得继续堆代码：

1. 页面需要同时消费 3 个以上接口并存在跨接口字段拼装。
2. 同一业务语义在两个以上页面各自实现且出现漂移。
3. 现有层次无法表达需求，导致需要跨层引用或绕过共享层。
4. bug 连续在同一模块重复出现，说明模块边界不合理。

提出优化建议时，至少说明：

1. 当前痛点与风险。
2. 拟调整的分层边界。
3. 对研发节奏和回归范围的影响。

---

### 小任务

可直接做，但必须说明：

- 改动依据
- 是否复用现有组件
- 最小验证结果

### 中任务

先明确：

1. 目标用户与操作场景
2. 页面状态变化
3. 影响文件
4. 是否需要新增共享组件
5. 验证方案

再开始改代码。

若任务属于试点支持项，还必须明确：

6. 当前外部契约是否已经稳定
7. 本轮是否会影响已有 smoke / visual gate

### 大任务 / 页面重做 / 多页联动

先补文档或方案，不要直接开写。

至少输出：

- 页面目标与非目标
- 交互草图或结构说明
- API 依赖
- 组件拆分
- 测试与验收口径
- 风险与待决策项

若属于试点页或试点支持任务，默认还应明确：

- 是否需要边界卡
- 是否需要定向 smoke 或视觉回归确认

## 视觉与交互默认规范

### 当前已确认前端基线

以下事项已确认，不再作为本阶段待决策：

- 视觉切换范围：仅新页面和重构页面切入新视觉，旧页面渐进迁移
- 主题模式：v1 仅正式支持亮色主题，暗色仅保留 token 预留
- 宽度策略：`1280+` 为正式工作宽度，`1024-1279` 为降级可用区间
- 涨跌色偏好：平台固定采用 A 股语义的红涨绿跌
- Shell 头部结构：保留品牌头部，但做克制化收敛

### 视觉方向

默认采用：

- 中性灰阶页面底色
- 白色或深浅对比分明的内容面板
- 深海军蓝品牌色
- 低阴影或无阴影
- 明确边框和间距秩序

默认不采用：

- 紫白默认审美
- 大面积渐变
- 玻璃拟态
- 大圆角消费级按钮
- 过量动效

### 排版与数据呈现

- 数字默认开启 `tabular-nums`
- 金额、数量、涨跌幅使用统一格式
- 数字列默认右对齐
- 空值统一显示 `—`
- 状态优先用共享 Badge，不靠整行染色

### 状态完整性

每个重要页面或容器至少明确：

- loading
- empty
- error
- loaded
- submit success / failure

不要只做“成功态”。

### 动效

- 默认仅保留操作反馈型动效
- 常规时长控制在 120-200ms
- 不做炫技动画
- 不做数据表 stagger 动画
- 不做数字滚动动画

## 测试与验证

对前端有效改动，默认验证顺序为：

1. `npm run typecheck`
2. `npm run check:rules`
3. `npm run test`
4. `npm run build`

更细的回归档位、截图基线刷新纪律与 CI 排查顺序，以 [frontend-regression-and-baseline-workflow-v1.md](/Users/congming/github/goldenshare/docs/frontend/frontend-regression-and-baseline-workflow-v1.md) 为准。

以下情况必须补测试或更新测试：

- 新增共享组件
- 变更共享组件行为
- 调整关键页面交互流程
- 修改鉴权、路由守卫、API 客户端
- 修复明确可复现的页面 bug

若本轮无法补足自动化测试，必须在结果中说明缺口。

## Code Review 关注点

前端评审默认先看以下问题：

1. 是否复用了共享组件，还是又在页面里拼了一份临时版本
2. 是否写死颜色、阴影、间距、圆角
3. 是否暴露了内部实现术语
4. 是否只有 happy path，没有空状态 / 错误态 / 加载态
5. 是否让页面文件继续膨胀
6. 是否为了快而绕开主题、类型、测试和共享层
7. 是否引入与当前视觉方向冲突的新风格

---

## 待决策处理规则

若任务涉及以下未最终拍板的问题，不要假装已定：

- 是否要改变已确认的亮色主题基线
- 是否要突破当前 `1280+` 正式工作宽度基线
- 是否要修改当前红涨绿跌的市场色语义
- 是否要把渐进迁移改为一次性全站视觉切换
- 是否要重做当前保留品牌头部的 Shell 结构

应在交付结果中单列：

- 已确认信息
- 默认假设
- `待决策`
- 若拍板 A / B 的影响
