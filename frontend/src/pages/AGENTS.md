# AGENTS.md — frontend/src/pages 路由页面规则

## 适用范围

本文件适用于 `frontend/src/pages/` 目录及其所有子目录。

若未来更深层目录存在更具体的 `AGENTS.md`，应同时遵守；发生冲突时，以更靠近当前目录的规则为准。

---

## 当前目录职责

`pages/` 是路由页面层，负责把用户任务真正落到页面上。

当前应承接的内容包括：

- 页面级查询组合
- 页面级状态流转
- URL / search param 与页面状态联动
- 共享组件编排
- 页面级文案与交互结构

---

## 当前硬约束

### 1. 页面是容器，不是新的基础设施层

页面可以做编排，但不应继续承接：

- 通用 API client 逻辑
- 全局主题规则
- 全局壳层
- 会被多个页面复用的 UI 标准件
- 会被多个页面复用的领域工具

### 2. 页面文件不应无限变厚

当前已有多页超过 800-1600 行，这是需要主动遏制的信号。

当页面出现以下情况时，应优先拆分：

- 单文件明显超过 400-600 行
- 同时承载筛选、表格、详情、表单、轮询
- 有两块以上可复用区块
- 同一页中出现大量本地帮助函数与状态派生

### 3. 页面可以有本地 helper，但不能假装共享

以下情况允许页面内相邻辅助文件：

- 只被 1 个页面使用
- 只被一组强绑定页面使用

若某段逻辑被多个页面复用，应迁出：

- UI 模式 -> `shared/ui`
- 通用格式化 / capability -> `shared/*`
- 领域状态 / workflow -> `features/*`

### 4. 运行时代码不要反向依赖 `app/**`

页面运行时代码默认不应 import：

- `app/router`
- `app/shell`
- `app/providers`

测试文件为了渲染环境可以引用：

- `app/theme`
- `AuthProvider`

---

## 推荐依赖方向

`pages/**` 允许依赖：

- `features/**`
- `shared/**`

`pages/**` 不应承接：

- `app/**` 的壳职责
- `shared/**` 的通用标准职责

---

## 当前目录的现实情况

当前 `pages/` 目录仍是平铺结构，且部分页面承载了过多逻辑。

这意味着：

- 目录级职责意识已经有了
- 但实现还没有真正完全收敛

因此当前阶段默认策略是：

- 先在既有目录下控厚度
- 再通过试点重构，把高频模式抽出
- 暂不做大规模 page 目录重排

---

## 页面改动默认要求

### 小改动

说明：

- 改动目标
- 是否复用现有共享组件
- 最小验证结果

### 中改动

说明：

1. 用户任务
2. 页面状态
3. 影响文件
4. 是否新增共享组件
5. 验证方案

### 大改动 / 重构

必须先有：

- 页面目标与非目标
- 改动边界
- 是否允许结构重构
- 回归范围
- 测试门禁

---

## 常见禁止事项

- 不要把通用表格模式继续写死在页面里
- 不要把新的 token / theme 规则直接写在页面 style 里
- 不要为了快，把领域逻辑塞进大页面本地函数
- 不要把内部对象术语直接当页面主文案
- 不要无边界地“顺手重构”

---

## 验证要求

改动 `pages/**` 后，至少考虑：

- `npm run typecheck`
- `npm run test`
- `npm run build`

若改动页面属于试点或关键流程，额外应确认：

- 页面可进入
- 主流程可完成
- 空状态 / 错误态 / 加载态可用
- 受影响页面的定向回归通过
