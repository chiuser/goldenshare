# AGENTS.md — frontend/src/pages 路由页面规则

## 适用范围

本文件适用于 `frontend/src/pages/` 目录及其所有子目录。

若未来更深层目录存在更具体的 `AGENTS.md`，应同时遵守；发生冲突时，以更靠近当前目录的规则为准。

---

## 当前目录职责

`pages/` 是路由页面层，负责把用户任务真正落到页面上。

当前应承接的内容包括：

- 页面级查询组合
- 页面级状态流转
- URL / search param 与页面状态联动
- 共享组件编排
- 页面级文案与交互结构

---

## 当前硬约束

### 1. 页面是容器，不是新的基础设施层

页面可以做编排，但不应继续承接：

- 通用 API client 逻辑
- 全局主题规则
- 全局壳层
- 会被多个页面复用的 UI 标准件
- 会被多个页面复用的领域工具

### 2. 页面文件不应无限变厚

当前已有多页超过 800-1600 行，这是需要主动遏制的信号。

当页面出现以下情况时，应优先拆分：

- 单文件明显超过 400-600 行
- 同时承载筛选、表格、详情、表单、轮询
- 有两块以上可复用区块
- 同一页中出现大量本地帮助函数与状态派生

### 3. 页面可以有本地 helper，但不能假装共享

以下情况允许页面内相邻辅助文件：

- 只被 1 个页面使用
- 只被一组强绑定页面使用

若某段逻辑被多个页面复用，应迁出：

- UI 模式 -> `shared/ui`
- 通用格式化 / capability -> `shared/*`
- 领域状态 / workflow -> `features/*`

### 4. 页面不得拼接后端原始 DTO

页面层禁止直接把后端原始字段“临时拼成业务结论”。

必须：

1. 通过 `features/entities` 提供的 ViewModel 渲染。
2. 保持 DTO -> ViewModel 映射单点收敛。
3. 页面只做编排和状态切换，不做字段语义拼接。

### 5. 禁止兼容方案与临时方案

页面开发中发现契约不合理时：

1. 先提契约收敛或结构优化方案。
2. 评审通过后再改代码。

不允许：

1. 先在页面写“兼容层”绕过契约问题。
2. 用临时分支逻辑长期留存。

### 6. 运行时代码不要反向依赖 `app/**`

页面运行时代码默认不应 import：

- `app/router`
- `app/shell`
- `app/providers`

测试文件为了渲染环境可以引用：

- `app/theme`
- `AuthProvider`

---

## 推荐依赖方向

`pages/**` 允许依赖：

- `features/**`
- `shared/**`

`pages/**` 不应承接：

- `app/**` 的壳职责
- `shared/**` 的通用标准职责

---

## 当前目录的现实情况

当前 `pages/` 目录仍是平铺结构，且部分页面承载了过多逻辑。

这意味着：

- 目录级职责意识已经有了
- 但实现还没有真正完全收敛

因此当前阶段默认策略是：

- 先在既有目录下控厚度
- 再通过试点重构，把高频模式抽出
- 暂不做大规模 page 目录重排

当前规模化推广期的默认策略是：

- 优先复用现有标准件，而不是继续在页面内手写重复模式
- 当前优先复用的标准件包括：
  - `AlertBar`
  - `FilterBar`
  - `TableShell`
  - `DetailDrawer`
  - `ActivityTimeline`
  - `TradeDateField`
  - `PriceText`
  - `ChangeText`

任务中心页面当前事实源：

- 任务记录列表、任务详情页、失败原因、进度节点、重试/停止动作均以 TaskRun view/list/summary API 为准。
- 页面不得重新拼接 `execution + steps + events + logs` 四套旧数据源。
- 失败原因主页面只展示一处，完整技术诊断只能按需打开详情。
- 成功态不展示失败原因或技术诊断入口。

## 页面改动默认要求

### 小改动

说明：

- 改动目标
- 是否复用现有共享组件
- 最小验证结果

### 中改动

说明：

1. 用户任务
2. 页面状态
3. 影响文件
4. 是否新增共享组件
5. 验证方案

若属于任务中心试点链路维护项或 `Phase 5/6` 的推广支持任务，还要说明：

6. 是否属于 `DataTable v1` 或 `TradeDateField v2`
7. 本轮边界卡是否已经明确

若属于 `Phase 6` 的页面推广批，还应说明：

8. 当前批次是否已在 [frontend-phase6-execution-plan-v1.md](/Users/congming/github/goldenshare/docs/frontend/frontend-phase6-execution-plan-v1.md) 中定义
9. 本轮是否触及已存在的 smoke / visual gate

### 大改动 / 重构

必须先有：

- 页面目标与非目标
- 改动边界
- 是否允许结构重构
- 回归范围
- 测试门禁

### 需求开发前置规则（新增）

中等及以上需求（新增页面、复杂交互、跨接口编排）必须先有详细技术方案并完成评审，评审通过后方可编码。

方案至少包含：

1. 页面目标与非目标。
2. 接口契约与字段口径。
3. ViewModel 组织方式。
4. 状态流转（loading/empty/error/stale）。
5. 验证与回归范围。

### bug 修复规则（新增）

bug 修复必须先说明：

1. 根因是什么。
2. 修复为什么合理。
3. 是否影响其他页面或共享组件。

禁止只做表层现象修补。

若改动的是试点页或试点支持任务，默认需要：

- 明确当前只接入哪些标准件
- 明确本轮不做哪些相邻重构
- 评估是否需要 smoke / visual gate 辅助回归

---

## 常见禁止事项

- 不要把通用表格模式继续写死在页面里
- 不要把新的 token / theme 规则直接写在页面 style 里
- 不要为了快，把领域逻辑塞进大页面本地函数
- 不要把内部对象术语直接当页面主文案
- 不要无边界地“顺手重构”
- 不要在页面里自行实现涨跌色、交易日判断、表格壳这类已经进入共享层的规则

---

## 代码规模门禁（新增）

1. 单文件超过 `400` 行必须评估拆分。
2. 单文件超过 `600` 行必须拆分，并在改动说明写清拆分边界。
3. 相同 UI/交互模式出现 3 次以上，必须抽象沉淀，禁止继续复制。

---

## 验证要求

改动 `pages/**` 后，至少考虑：

- `npm run typecheck`
- `npm run test`
- `npm run build`

若改动页面属于试点或关键流程，额外应确认：

- 页面可进入
- 主流程可完成
- 空状态 / 错误态 / 加载态可用
- 受影响页面的定向回归通过

若改动页面属于高可见试点页，还应评估：

- `npm run test:smoke`
- 是否影响现有视觉基线

页面改动的最小验证档位、截图基线更新纪律与 CI 失败排查顺序，以 [frontend-regression-and-baseline-workflow-v1.md](/Users/congming/github/goldenshare/docs/frontend/frontend-regression-and-baseline-workflow-v1.md) 为准。
