# AGENTS.md — frontend/src/app 组合根规则

## 适用范围

本文件适用于 `frontend/src/app/` 目录及其所有子目录。

若未来更深层目录存在更具体的 `AGENTS.md`，应同时遵守；发生冲突时，以更靠近当前目录的规则为准。

---

## 当前目录职责

`app/` 是前端应用壳与组合根，对应的是“应用如何被装起来”，而不是“具体业务页面怎么实现”。

当前应承接的内容包括：

- Provider 装配
- Router 定义与路由挂载
- 鉴权壳与布局壳的拼装
- App shell
- Theme / token
- 全局级通知、全局查询客户端配置

---

## 当前硬约束

### 1. `app` 不是页面业务层

不要把以下内容继续放进 `app`：

- 页面级查询逻辑
- 页面级筛选状态
- 页面级表单处理
- 业务格式化逻辑
- 与单个页面绑定的交互细节

### 2. `router.tsx` 只负责路由与守卫

允许：

- 路由树定义
- 基础重定向
- 鉴权守卫
- Layout 装配

不允许：

- 直接写页面业务逻辑
- 直接写接口拼装
- 在 router 中堆积大量页面专用帮助函数

### 3. `shell.tsx` / `share-shell.tsx` 只负责壳层

允许：

- 导航结构
- 全局头部
- 侧边栏
- 全局用户信息展示

不允许：

- 页面业务规则
- 数据集级或任务级状态流转
- 具体页面专用的临时逻辑

### 4. `theme.ts` 是唯一主题入口

新的视觉 token、组件默认样式覆盖、全局主题约束，应优先进入 `theme.ts` 或其后续拆出的主题文件。

不要：

- 在页面里到处复制 Mantine 覆盖
- 在多个地方重复定义品牌色、圆角、阴影、间距

---

## 允许依赖

`app/**` 可以依赖：

- `app/**`
- `features/**`
- `shared/**`
- `pages/**`（仅路由挂载与页面入口装配）

`app/**` 不应被：

- `pages/**`
- `features/**`
- `shared/**`

在运行时代码中反向依赖。

说明：

- 页面测试文件为了构造渲染环境，可以引用 `app/theme` 或 `app/providers`
- 运行时代码默认不要从 `pages/**` 反向 import `app/**`

---

## 推荐工作方式

### Provider 变更

若调整：

- `QueryClient`
- `MantineProvider`
- `Notifications`
- 全局 Context

必须说明：

- 影响范围
- 默认行为变化
- 是否影响所有页面

### Router 变更

若调整：

- 路由结构
- Redirect 规则
- Layout 组织
- 鉴权守卫

必须说明：

- 影响路由
- 是否影响深链
- 是否影响登录态恢复

### Shell 变更

若调整壳层：

- 顶栏
- 侧栏
- 导航分组
- 全局版本/用户区域

必须优先考虑：

- 是否属于壳层职责
- 是否会扩大到页面业务逻辑
- 是否需要先有专项方案

---

## 需要警惕的信号

出现以下信号时，说明 `app` 目录开始失控：

- `router.tsx` 继续变成巨型业务文件
- `shell.tsx` 里出现任务、数据集、表单、表格逻辑
- `theme.ts` 被页面内 style 覆盖绕开
- 为了快，把页面逻辑“临时放在 app 里”

---

## 验证要求

改动 `app/**` 后，至少考虑：

- `npm run typecheck`
- `npm run test`
- `npm run build`

若影响路由或壳层，额外确认：

- 登录页
- 鉴权跳转
- 主壳入口
- 受影响页面的挂载路径
