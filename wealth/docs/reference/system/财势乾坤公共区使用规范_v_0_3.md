# 财势乾坤项目公共区使用规范 v0.3

> 文档角色：多角色协作公共区规范  
> 适用对象：产品与 UI 总控、01 Design Token、02 页面原型与 HTML Showcase、03 组件库、04 API 契约与数据字典、05 Codex 落地提示词、Codex 实现线程  
> 当前状态：Review 草案  
> 维护责任：产品与 UI 总控  
> 本版修订重点：确认 Google Drive 公共区已新增 `review` 与 `codex` 目录，并将 `02-market-overview-page-design.md` 纳入“设计”目录的正式页面设计基线。

---

## 1. 规范目的

财势乾坤项目采用多个角色协同工作。为了避免 01～05 号角色在不同会话中各自理解、各自生成、上下文不一致，必须建立统一的公共区读取与产出规则。

公共区的核心作用是：

1. 让所有角色读取同一套上游文档；
2. 让所有角色把产出放回同一套目录；
3. 让产品与 UI 总控可以审计文档之间是否冲突；
4. 让 Codex 能基于稳定文档执行前端落地；
5. 避免页面设计、组件、API、Showcase、Codex 提示词互相脱节。

---

## 2. 当前公共区结构

当前项目公共区以 Google Drive 中的 **“财势乾坤”** 目录为主。

当前已确认一级目录如下：

```text
财势乾坤/
├── 项目总说明/
├── 产品文档/
├── 设计/
├── 数据字典与API文档/
├── showcase/
├── review/
├── codex/
└── Logo/
```

另有数据源参考目录：

```text
tushare接口文档/
├── README.md
├── docs_index.csv
└── docs/sources/tushare/...
```

### 2.1 财势乾坤目录用途

`财势乾坤/` 是项目协作主公共区，用于存放：

- 项目总说明；
- 页面级 PRD；
- Design Token；
- 组件规范；
- 页面设计文档；
- API 与数据字典；
- HTML Showcase；
- Review 变更单；
- Codex 落地提示词；
- Logo 等品牌资产。

### 2.2 tushare接口文档目录用途

`tushare接口文档/` 只作为 API 和数据字典设计的源数据参考区。

使用原则：

1. 只记录 Tushare 源站接口说明和字段事实；
2. 可用于字段口径、单位、更新时间、输入输出参数校验；
3. 不直接作为财势乾坤前台 API 的形态；
4. 财势乾坤 API 必须以业务对象、页面模块和前端使用场景组织。

## 3. 目录职责

### 3.1 项目总说明

路径：

```text
财势乾坤/项目总说明/
```

职责：

- 存放项目顶层总控文档；
- 约束产品定位、系统边界、页面清单、P0 范围、UI 原则、API 原则、Codex 协作原则；
- 所有角色必须优先读取。

当前基线文件：

```text
财势乾坤行情软件项目总说明_v_0_2.md
```

### 3.2 产品文档

路径：

```text
财势乾坤/产品文档/
```

职责：

- 存放页面级 PRD；
- 位于项目总说明和具体页面设计之间；
- 约束页面目标、页面归属、功能范围、非目标、用户任务、模块清单、交互边界、状态需求和下游角色要求。

当前市场总览基线文件：

```text
市场总览产品需求文档 v0.2.md
```

注意：市场总览页面正式名称为 **市场总览**，属于 **乾坤行情**，不是独立一级菜单。

### 3.3 设计

路径：

```text
财势乾坤/设计/
```

职责：

- 存放 Design Token；
- 存放组件规范；
- 存放页面设计文档；
- 存放可供 02、03、05 使用的视觉与交互设计约束。

当前已识别文件：

```text
02-market-overview-page-design.md
03-design-tokens.md
04-component-guidelines.md
```

使用规则：

1. `02-market-overview-page-design.md` 是市场总览页面设计文档，承接 PRD，并指导 Showcase；
2. `03-design-tokens.md` 是视觉规范与 Design Token；
3. `04-component-guidelines.md` 是组件库与交互组件方案；
4. 02、03、05 处理市场总览相关任务时，必须读取该目录下的上述文件；
5. 如页面设计文档更新，应同步评估是否影响 Token、组件、API 和 Codex 提示词。

### 3.4 数据字典与API文档

路径：

```text
财势乾坤/数据字典与API文档/
```

职责：

- 存放 P0 数据字典；
- 存放页面级 API 草案；
- 存放字段映射、字段单位、更新频率、异常状态、Mock 数据建议；
- 供 02 页面设计、03 组件 Props、05 Codex 提示词使用。

当前已识别文件：

```text
p0-data-dictionary-v0.3.md
market-overview-api-v0.3.md
p0-data-dictionary-v0.2.2.md
market-overview-api-v0.2.1.md
p0-data-dictionary-v0.2.1.md
```

使用规则：

1. 默认优先读取最高版本；
2. 历史版本保留用于追溯；
3. 如果同一主题存在多个版本，角色必须说明实际采用哪个版本；
4. API 文档必须保持市场总览的客观事实边界，不得把市场温度、市场情绪指数、资金面分数、风险指数混入市场总览核心展示字段。

### 3.5 showcase

路径：

```text
财势乾坤/showcase/
```

职责：

- 存放 HTML Showcase；
- 存放可供 Codex 和前端工程师还原的单文件 HTML/CSS/JS；
- 存放冻结参考版。

建议命名：

```text
market-overview-v1.html
market-overview-v1.1.html
market-overview-v1.2.html
```

当某一版被产品总控确认后，应标记为：

```text
market-overview-v1.freeze.html
```

或在 Review 变更单中注明“冻结版本”。

### 3.6 review

路径：

```text
财势乾坤/review/
```

职责：

- 存放 Review 变更单；
- 记录用户反馈、问题位置、影响范围、分发角色和处理状态；
- 作为 UI 修改、API 修改、组件修改和 Codex 提示词修改的中间依据。

建议文件：

```text
market-overview-html-review-v1.md
market-overview-html-review-v2.md
```

Review 变更单必须记录：

1. 用户反馈；
2. 问题位置；
3. 变更类型；
4. 影响范围：PRD / Token / 组件 / API / Showcase / Codex；
5. 涉及角色；
6. 本轮是否接受修改；
7. 新 Showcase 版本号；
8. 是否需要更新 Codex 提示词。

### 3.7 codex

路径：

```text
财势乾坤/codex/
```

职责：

- 存放 Codex 落地提示词；
- 存放 Codex smoke test 要求；
- 存放文件范围、禁止修改范围、实现验收标准；
- 只在 PRD、页面设计、Token、组件、API、Showcase 基本稳定后生成。

建议文件：

```text
market-overview-codex-prompt-v1.md
market-overview-codex-prompt-v1.1.md
```

Codex 提示词生成前，必须确认：

1. 市场总览 PRD 已确认；
2. 页面设计文档已确认；
3. Design Token 已可用；
4. 组件规范已可用；
5. API 草案已可用；
6. Showcase 已进入候选冻结状态。

### 3.8 Logo

路径：

```text
财势乾坤/Logo/
```

职责：

- 存放品牌 Logo、图标、透明背景版本、应用图标等品牌资产；
- 不作为 P0 页面设计的必读文档，除非本轮涉及品牌视觉或顶部栏 Logo。

## 6. 文件版本规则

### 6.1 版本号

建议使用：

```text
v0.1
v0.2
v0.3
v1
v1.1
```

规则：

- `v0.x`：Review 草案；
- `v1`：可以进入实现的候选稳定版；
- `v1.freeze`：产品总控确认的冻结参考版；
- `v1.1`：冻结后的局部修订。

### 6.2 多版本文件处理

如果同一目录存在多个版本，默认读取最高版本。

示例：

```text

```

默认采用：

```text

```

但角色必须在输出开头声明实际采用版本。

## 8. 市场总览当前基线

市场总览当前基线规则如下：

1. 页面名称：市场总览；
2. 所属系统：乾坤行情；
3. 不是独立一级菜单；
4. 可作为系统默认落地页；
5. 桌面端不使用固定 SideNav；
6. 桌面端采用 TopMarketBar + Breadcrumb + PageHeader + ShortcutBar + 全宽行情内容区；
7. 页面只展示 A 股市场客观事实；
8. 不展示市场温度、市场情绪指数、资金面分数、风险指数作为首页核心结论；
9. 中国市场红涨绿跌；
10. 视觉风格专业、沉稳、高密度、有金融终端感。

所有角色在处理市场总览相关任务时，必须遵守以上基线。

## 9. 变更流程

任何页面设计变更，都应按以下流程处理。

```text
用户提出 Review 意见
  ↓
产品与 UI 总控整理 Review 变更单
  ↓
判断影响范围：PRD / Token / 组件 / API / Showcase / Codex
  ↓
分别给 01～05 号角色分发任务
  ↓
角色修改产物并放回公共区
  ↓
产品与 UI 总控审计
  ↓
确认 Showcase 冻结版
  ↓
05 生成 Codex 落地提示词
  ↓
Codex 实现
```

---

## 10. 变更分级

| 类型 | 示例 | 处理方式 |
|---|---|---|
| A. 纯视觉调整 | 字号、间距、边框、颜色深浅 | 02 修改 Showcase，必要时 01 更新 Token |
| B. 组件结构调整 | 指数卡结构、榜单表格列、连板天梯布局 | 03 更新组件规范，02 修改 Showcase |
| C. API/数据调整 | 新字段、新图表、新筛选、新排序 | 04 更新 API 和数据字典，02 更新 Mock |
| D. 产品范围调整 | 增删模块、改页面职责、改导航模式 | 产品与 UI 总控先更新 PRD，再分发任务 |
| E. Codex 落地调整 | 文件范围、路由、实现约束、Smoke Test | 05 更新 Codex 提示词 |

## 11. Tushare 文档使用规范

`tushare接口文档` 只作为数据源接口说明和字段口径参考。

使用原则：

1. 可以引用 Tushare 原接口字段；
2. 可以做 `Tushare 原字段 → 财势乾坤业务字段` 映射；
3. 可以记录单位归一化规则；
4. 不能直接复刻 Tushare API 形态作为前台业务 API；
5. 不能把 Tushare 文档中的接口限制直接暴露给前端页面；
6. 财势乾坤 API 必须以页面业务对象组织。

示例：

```text
Tushare: daily.amount 单位为千元
财势乾坤业务字段：turnoverAmount，单位统一为元
前端显示：自动格式化为亿元/万亿元
```
