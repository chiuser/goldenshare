# 前端交付流程规范 v1

> 角色说明：本文件是“前端交付流程专题文档”。  
> 当前前端强约束与统一门禁请以 [frontend-current-standards.md](/Users/congming/github/goldenshare/docs/frontend/frontend-current-standards.md) 为准。

## 1. 文档目的

本文用于把 `goldenshare` 当前前端开发从“页面能做出来”升级为“需求、交互、实现、测试、验收可复用的交付流程”。

适用范围：

- `frontend/**`
- 与前端页面直接相关的文档协作
- 前端页面对应的验收与上线准备

本文不替代具体页面方案，也不替代后端架构规则；它只定义前端交付的默认路径和门禁。

---

## 2. 当前阶段目标

当前前端工作的优先级如下：

1. 保护 `Phase 2-4` 已收敛下来的页面与组件成果
2. 深化自动化门禁与回归流程
3. 保持组件目录、Showcase、测试与 AGENTS 同步
4. 再逐步推广到相邻页面与更多链路

当前回归与截图基线更新纪律请以 [frontend-regression-and-baseline-workflow-v1.md](/Users/congming/github/goldenshare/docs/frontend/frontend-regression-and-baseline-workflow-v1.md) 为准。

因此，当前阶段不追求：

- 一次性推翻全部已有页面
- 为了追求“看起来更炫”而脱离业务任务
- 为单个页面临时发明一套自己的流程
- 在没有回归保护的前提下继续扩大页面改造面

## 3. 标准交付阶段

每个中等及以上前端任务，默认按以下阶段推进。

### 3.1 阶段 0：任务 intake

先回答 4 个问题：

1. 这是修 bug、增量功能，还是页面重做
2. 用户是谁
3. 用户想完成的动作是什么
4. 这次改动的成功标准是什么

产出：

- 一段目标描述
- 影响页面
- 成功口径

### 3.2 阶段 1：需求与用户任务梳理

必须把需求改写成用户动作，而不是内部对象列表。

至少说明：

- 用户角色
- 使用场景
- 主任务
- 次任务
- 不在本轮处理的内容

推荐格式：

```md
- 用户角色：
- 主任务：
- 次任务：
- 本轮不做：
```

### 3.3 阶段 2：交互与页面状态设计

在写代码前，至少明确以下页面状态：

- 首次加载
- 空状态
- 加载失败
- 有数据但存在异常提示
- 用户提交中
- 用户提交成功 / 失败

同时明确页面结构：

- PageHeader
- 筛选区 / 操作区
- 主内容区
- 详情区 / 抽屉 / Modal
- 异步任务反馈路径

若页面复杂，应输出一份简版页面说明。

建议模板：

```md
## 页面结构
- 顶部：
- 主区域：
- 次区域：

## 状态
- loading：
- empty：
- error：
- submit success：
- submit failure：
```

### 3.4 阶段 3：交互评审

交互评审至少检查以下内容：

1. 页面是否围绕用户任务组织，而不是后端对象组织
2. 主操作是否清晰
3. 重要信息是否在首屏
4. 筛选和表格是否互相抢注意力
5. 是否存在只有开发者才看得懂的术语
6. 是否补齐了空状态、错误态、加载态
7. 是否已有共享组件可复用

未过评审前，不进入大规模编码。

### 3.5 阶段 4：前端技术方案设计

编码前，至少明确：

- 影响文件
- 新增共享组件
- 页面内子模块拆分
- API 依赖
- URL 状态
- 测试计划

对复杂页面，建议把工作拆成 4 条并行线：

1. 需求与交互
2. token 与共享组件
3. 页面实现
4. 测试与验收

每条线都应有明确 owner 和写入范围，避免多人同时在一个超大页面里互相覆盖。

### 3.6 阶段 5：实现

实现时遵守以下默认顺序：

1. 先补 theme / token / 共享组件
2. 再搭页面骨架
3. 再接数据和交互
4. 最后补细节文案和反馈

不要反过来先把业务全塞进一个页面文件，再回头“提炼组件”。

### 3.7 阶段 6：测试与验收

默认门禁：

1. `npm run typecheck`
2. `npm run check:rules`
3. `npm run test`
4. `npm run build`

验收至少覆盖：

- 页面可进入
- 主流程可完成
- 失败可反馈
- 空状态可理解
- 文案对非开发者可读

若改动涉及高可见共享组件、任务中心试点页或 smoke 覆盖页，还应按 [frontend-regression-and-baseline-workflow-v1.md](/Users/congming/github/goldenshare/docs/frontend/frontend-regression-and-baseline-workflow-v1.md) 评估是否需要：

- `npm run test:smoke`
- 刷新截图基线
- 在交付说明中解释截图基线更新原因

### 3.8 阶段 7：上线准备

上线前需要有简版变更说明：

- 影响页面
- 主要变更
- 风险点
- 验收结果
- 若失败如何回退

若涉及明显交互变化，还应同步页面说明文档或交付记录。

## 4. 必要产物

### 小任务

至少有：

- 目标说明
- 改动文件
- 验证结果

### 中任务

至少有：

- 目标理解
- 页面状态说明
- 技术方案摘要
- 测试方案
- 待决策项

### 大任务 / 页面重做

至少有一份“前端交付简报”，建议结构如下：

```md
# 前端交付简报

## 1. 背景与目标
## 2. 用户角色与主任务
## 3. 页面结构
## 4. 状态设计
## 5. API 与数据依赖
## 6. 共享组件与 token 影响
## 7. 测试与验收
## 8. 风险与待决策
```

## 5. 交互评审清单

进入编码前，建议逐项检查：

- [ ] 页面标题、按钮、说明文案面向用户任务，而不是内部对象
- [ ] 用户能在首屏知道“这页主要做什么”
- [ ] 主操作只有一个中心
- [ ] 页面不存在无意义的品牌色装饰
- [ ] 空状态、错误态、加载态齐全
- [ ] 数据格式、单位、时间格式统一
- [ ] 日期输入和筛选模式复用共享组件
- [ ] 表格密度、对齐、状态展示遵守统一规范
- [ ] 已确认是否需要新增共享组件
- [ ] 已确认是否需要补测试

---

## 6. 技术方案评审清单

- [ ] 是否复用现有共享组件
- [ ] 是否把 page-only 样式写成了共享标准
- [ ] 是否把 transport、业务逻辑、展示逻辑混在一起
- [ ] 是否引入新的视觉语言冲突
- [ ] 是否继续让超大页面文件膨胀
- [ ] 是否补充类型定义和错误处理
- [ ] 是否明确了 URL 状态与页面状态关系

---

## 7. 多线程协同默认分工

为解决“大家都在做前端，但效率低、互相打架”的问题，默认采用下列分工模型：

### 角色 A：需求与交互 owner

负责：

- 需求收敛
- 页面结构
- 状态设计
- 待决策整理

### 角色 B：设计 token / 共享组件 owner

负责：

- 主题 token
- 共享 UI 组件
- 视觉一致性

### 角色 C：页面实现 owner

负责：

- 页面编排
- 查询与交互
- 容器级状态处理

### 角色 D：测试与验收 owner

负责：

- 页面测试
- 回归验证
- 验收记录

默认要求：

- 每个 owner 都有明确写入范围
- 未经同步，不跨范围大改别人正在收敛的区域
- “共享组件”优先于“页面私有实现”

## 8. 待决策管理

若任务遇到未拍板事项，必须显式记录，而不是边做边默认。

建议统一写成：

```md
## 待决策

### D1. 标题 / 方向
- 选项 A：
- 选项 B：
- 默认假设：
- 影响：
```

当前前端治理已确认基线如下：

1. v1 仅正式支持亮色主题，暗色只做 token 预留
2. `1280+` 为正式工作宽度，`1024-1279` 降级可用
3. 平台固定采用红涨绿跌，不提供“绿涨红跌”用户偏好切换
4. 旧视觉风格采用“新页面 / 重构页面优先”的渐进收敛方式
5. Shell 保留品牌头部结构，但整体风格收敛到更克制的版本

若具体任务要偏离上述基线，应单独列为本任务的 `待决策`，不要默认继承旧分歧。
