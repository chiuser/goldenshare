# 行情业务系统 Web API 契约规范 v1

## 1. 目标

定义业务页面与后端的 API 协议规范，保证：

1. 前后端语义稳定。
2. 字段口径一致。
3. 错误处理统一。

业务主接口基线：

1. `docs/platform/quote-detail-api-spec-v1.md`

---

## 2. 路由边界

业务页面只消费：

1. `/api/v1/quote/*`
2. `/api/v1/market/*`
3. `/api/v1/auth/*`（鉴权相关）

不直接消费：

1. `/api/v1/ops/*`（运维接口）

---

## 3. 请求规范

1. 所有请求通过 `shared/api/client`。
2. 鉴权请求统一携带 `Authorization: Bearer <token>`。
3. 所有查询参数通过 typed schema 封装，不在页面拼 URL。

---

## 4. 响应规范

1. 字段命名统一 `snake_case`。
2. 时间字段统一格式（`YYYY-MM-DD` 或明确 datetime 格式）。
3. 数值字段统一类型约定（价格、涨跌幅、成交额等）。

页面消费规则：

1. 原始 DTO 不直接进组件。
2. 必须经 `entities` 映射为 ViewModel。

---

## 5. 错误规范

统一错误结构（示意）：

```json
{
  "detail": {
    "code": "UNSUPPORTED_PERIOD",
    "message": "当前能力暂不支持该周期"
  }
}
```

前端必须按 `code` 分流，不靠字符串匹配。

必备处理：

1. `401`: 会话失效，触发登录恢复流程。
2. `403`: 权限不足，展示权限态页面。
3. `4xx`: 参数错误，展示可读提示。
4. `5xx`: 系统异常，展示重试入口和 request_id（若有）。

---

## 6. 版本与变更策略

1. 新字段增加：向后兼容。
2. 字段语义变更：必须文档先行并做消费者审计。
3. 删除字段：必须走迁移期，不允许静默删除。

---

## 7. 联调清单

1. 接口可达性检查。
2. 授权/未授权路径检查。
3. 关键参数边界值检查。
4. 空数据与滞后数据渲染检查。
5. 错误码渲染一致性检查。
