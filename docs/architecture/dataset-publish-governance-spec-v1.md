# 数据集发布治理规范 v1（Raw -> Std -> Serving）

> 目标：把“字段定义、跨源 mapping、清洗、入 std、发布 serving”做成强约束流程。  
> 结论：**缺任一项，不允许进入开发阶段。**
>
> 角色说明：本文件是“数据发布治理专题文档”。  
> 子系统边界与统一强约束请以 [foundation-current-standards.md](/Users/congming/github/goldenshare/docs/architecture/foundation-current-standards.md) 为准。

## 1. 适用范围

- 所有新增数据集。
- 所有既有数据集的“新增来源/新增字段/改发布策略”变更。
- 单源数据集也必须遵守（只是在“跨源规则”处标记 N/A）。

## 2. 开发前门禁（必须全部完成）

以下 6 项任一未完成，禁止编码：

1. 接口契约冻结：接口文档链接、抓取日期、返回字段清单固定。
2. 字段字典冻结：标准字段定义（类型/精度/单位/空值策略）完成。
3. Mapping 矩阵冻结：`source -> std` 字段映射、归一规则、缺失处理完成。
4. 清洗规则冻结：异常值、重复值、日期格式、枚举归一规则完成。
5. 发布规则冻结：`std -> serving` 的来源优先级、冲突规则、去重键完成。
6. 测试契约冻结：最少测试集（单测+集成+回归）列全。

## 3. 五层职责边界（不可混层）

1. Source（接口层）
- 只负责“拉数据”，不做业务语义决策。

2. Raw（保真层）
- 尽可能保留来源原貌。
- 允许最小格式归一（例如日期字符串转 date），禁止业务推断填充。

3. Std（标准层）
- 唯一允许做字段对齐、单位归一、可复用清洗规则的层。
- 多源必须保留 `source_key`。

4. Serving（服务层）
- 面向上层统一口径。
- 只保留对上需要字段。
- 必须可追溯来源（例如 `resolved_source_key` / `policy_version`）。

5. Biz API（业务接口层）
- 只读 serving，禁止直接读 raw/std。

## 4. 字段定义规范（Std 字段字典）

每个数据集必须有“标准字段字典”，至少包含：

- `field_name`
- `business_name`
- `type / precision / scale`
- `unit`
- `nullable`
- `default`
- `semantic_note`（口径说明）

禁止行为：

- 仅按数据库默认类型“先落再说”。
- 未定义单位就混用（例如万元/元，手/股）。

## 5. 跨源 Mapping 规范（Source -> Std）

必须输出“字段对齐矩阵”，每个 std 字段逐项定义：

- tushare 来源字段
- biying 来源字段
- 归一函数（例如交易所编码映射、空白字符清洗）
- 缺失策略（保留空值 / 常量回填 / 回退来源）
- 冲突说明（双源都有但值不一致时如何处理）

关键原则：

- **相同/不同的定义，只能基于“双方都具备语义可比”的字段。**
- 不可比字段不得参与“差异判定”（例如某源根本不提供）。

## 6. 清洗规范（Raw -> Std）

清洗必须可声明、可测试、可复用：

- 日期归一：明确时区与格式。
- 枚举归一：建立标准码表（例如 `SZSE -> SZ`）。
- 字符串归一：trim、全角半角、连续空格处理策略。
- 异常值处理：超界值、非法枚举、空主键处理。
- 重复处理：定义唯一键和覆盖策略（最后写入/按时间/按版本）。

禁止行为：

- 在服务代码里散落 `if/else` 临时修补且无文档。
- 清洗规则无测试直接上线。

## 7. Std 入库规范

- 主键必须显式定义（多源通常是 `source_key + 业务键`）。
- Upsert 冲突键必须与文档一致。
- 必须声明“批内重复如何去重”。
- 必须声明“更新列范围”（哪些列允许覆盖）。

## 8. Serving 发布规范（Std -> Serving）

每个数据集必须声明发布策略对象，至少包含：

- 发布主键（例如 `ts_code, trade_date`）
- 来源优先级（例如 `tushare > biying`）
- 字段级冲突策略（可选：全行优先或字段级优先）
- 回退策略（主源缺失时是否用次源）
- 不可逆行为（删除/截断）策略
- 追溯字段（来源、策略版本、发布时间）

最低要求：

- 任何发布都必须可追溯“这条 serving 来自哪个 source / 哪个策略版本”。

## 9. 变更控制规范（新增字段/新增来源）

1. 新增来源：
- 先补 mapping 矩阵与清洗规则，再接入同步。
- 不允许“先写库后补文档”。

2. 新增字段：
- 先改 Std 字段字典，再改 serving 字段定义，再改 API 契约。
- 必须给出兼容策略（默认值/空值/版本说明）。

3. 规则变更：
- 必须升策略版本，并记录影响范围。

## 10. 最小测试集规范（必须）

1. 单元测试
- mapping 测试（含可比字段和不可比字段）
- 清洗规则测试（异常值/空值/枚举归一）
- upsert 冲突测试
- 发布策略测试（优先级、回退、字段覆盖）

2. 集成测试
- 一次完整链路：source -> raw -> std -> serving
- Ops 可观测性（任务状态、进度、结果）

3. 回归测试
- 至少验证同域一个旧数据集不受影响。

## 12. Stock Basic 示例（当前项目口径）

仅示例“规范如何写”，不是唯一实现：

- 可比字段：`ts_code`, `name(归一后)`, `exchange(归一后)`
- 不可比字段：某一来源独有字段（例如行业/英文名等）
- 发布策略：同 `ts_code` 只保留一条 serving；主源优先；次源仅补缺
