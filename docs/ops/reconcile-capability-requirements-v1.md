# 多源对账能力需求文档 v1

更新时间：2026-04-12

## 1. 背景

多源改造后，数据质量风险从“单源是否更新”升级为“多源是否一致、融合是否跑偏”。  
对账能力必须成为独立能力，而不是临时 SQL。

目标：

- 可重复执行
- 可追溯
- 可设置门禁阈值
- 后续可接 Ops 展示与人工纠偏

## 2. 总体需求（完整形态）

### 2.1 能力范围

1. 对账规则管理（规则版本化）
2. 对账运行管理（每次运行留痕）
3. 对账结果明细（可筛选）
4. 异常工单化（可跟踪处理）
5. 人工纠偏审计（可回放）

### 2.2 核心对象

- `reconcile_rule`：规则定义（数据集、来源、可比字段、归一规则、阈值）
- `reconcile_run`：一次运行实例（状态、摘要、耗时、错误）
- `reconcile_result`：明细差异
- `reconcile_issue`：可操作问题单
- `reconcile_fix_action`：人工修复操作审计

### 2.3 设计原则

1. 相同/不同的定义必须基于“双方都具备语义可比”的字段。
2. 不可比字段不得参与差异判定。
3. 对账口径必须文档化并可测试。
4. 阈值门禁可用于发版流程（超阈值返回非0）。

## 3. 第一阶段（当前落地范围）

> 先实现“半天可落地”的命令行门禁工具，用于防止开发跑偏。

### 3.1 命令

- `goldenshare reconcile-stock-basic`

### 3.2 固定口径

- 主键：`ts_code`
- 可比字段：
  - `name_norm`（去空白后比较）
  - `exchange_norm`（交易所编码归一后比较）

### 3.3 输出分类

- `only_tushare`
- `only_biying`
- `comparable_diff`

### 3.4 输出内容

1. 汇总计数（总并集、可比数量、三类差异数量）
2. 每类前 N 条样例（可配置）
3. 阈值校验结果（超阈值返回非0）

### 3.5 阈值参数

- `--threshold-only-tushare`
- `--threshold-only-biying`
- `--threshold-comparable-diff`
- `-1` 表示关闭该项门禁

## 4. 使用场景

1. 本地开发回归
2. 发版前卡口检查
3. 多源规则变更后的基线校验

## 5. 后续演进（不在本次实现）

1. 对账结果持久化入库（run/result/issue）
2. 自动调度运行（每日/每周）
3. Ops 页面展示与人工纠偏入口
4. 扩展到 `equity_daily_bar / adj_factor / daily_basic`
