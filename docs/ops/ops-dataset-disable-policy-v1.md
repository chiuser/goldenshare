# 运维数据集停用策略 v1（过渡方案）

## 背景

当前存在个别数据集需要“保留同步能力，但不纳入健康度观测”的需求。  
这类需求本质上属于**运维控制面配置**，理想做法应由配置表驱动，而不是硬编码。

考虑当前上线节奏与风险控制，先采用代码级停用策略，满足业务需求并保持改动可控。

## 目标

1. 数据状态页可明确展示“已停用”状态。  
2. 总览状态概览新增“已停用”统计。  
3. 已停用数据集不计入每日健康度观测（不进入滞后/严重滞后/未知统计）。  
4. 同步任务保留，不影响手动/自动执行链路。

## 当前实现边界（v1）

1. 停用名单由代码常量维护（非数据库配置）。  
2. 仅影响“观测与展示”口径，不影响执行能力。  
3. 前端不提供“启用/停用”交互按钮。  

## 设计说明

1. 在新鲜度计算阶段，将命中停用名单的数据集状态强制为 `disabled`。  
2. `disabled` 在状态页和总览中可见，但不纳入健康度异常统计。  
3. 日报中增加停用数量展示，并将停用项从重点告警逻辑中排除。  

## 为什么说这不是长期最佳方案

代码级名单存在以下问题：

1. 变更需要发版，无法快速运营调整。  
2. 缺少审计链路（谁在何时停用/恢复）。  
3. 多环境一致性依赖部署流程，容易出现口径漂移。

## 后续演进方向（v2）

后续升级为配置化控制：

1. 引入 `ops.dataset_control` 表（`dataset_key`、`is_disabled`、`reason`、`updated_at`）。  
2. 新鲜度服务读取控制表覆盖状态。  
3. 结合权限模型开放“停用/恢复”操作并记录审计日志。  

## 当前默认停用数据集

当前无默认停用数据集。
